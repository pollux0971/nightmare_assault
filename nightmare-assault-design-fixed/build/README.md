# build/ · AI 開發工具包

> 這個資料夾裝的是「讓 AI 穩定實作這個系統」的工具。design 那 9 份文件講「系統是什麼」，這裡講「怎麼讓 AI 一塊一塊把它蓋出來而不失憶」。

---

## 為什麼需要這個資料夾

你的系統有 8 個 agent、雙層 bible、並行 patch——是中大型系統。若把整包設計丟給 AI 說「實作吧」，它會失憶：第 3 份文件的 schema，到第 7 次對話它忘了；它會重新發明你定義過的東西，寫出前後矛盾的代碼。

**文件再具體都救不了這件事**——問題不在文件，在 AI 上下文窗口裝不下整個專案。

解法是把實作拆成「AI 一次吃得下的工單」，並在每次寫代碼時餵它「不可變接口」當錨點。

---

## 四件產出

| 檔案 | 作用 | 何時用 |
|------|------|--------|
| `CONTRACTS.md` | 不可變接口（資料類、函式簽名、分隔符、目錄） | **每次**寫代碼都貼相關段落 |
| `BUILD-PLAN.md` | 19 個工單，有依賴順序、可獨立測試 | 決定「現在做哪個工單」 |
| `CHECKLIST.md` | 實作易錯點，可勾選、有驗收 | 做工單時對照確認 |
| `DESIGN-CHANGES.md` | 為 AI 施工而做的設計調整摘要 | 開工前讀一次，理解大方向 |

---

## 標準工作流程（每個工單重複）

```
1. 開新對話（或清理上下文），避免累積失憶
2. 對 AI 說：「做工單 N」
3. 貼上：
   - BUILD-PLAN.md 裡工單 N 的段落
   - CONTRACTS.md 裡相關的資料類/簽名
   - CHECKLIST.md 裡對應的檢查項
   - （若工單牽涉某 agent，附該 skills/xxx/SKILL.md）
4. AI 寫實作 + 測試
5. 你跑測試，對照工單驗收標準與 checklist
6. 過了 → commit，進工單 N+1
   沒過 → 把錯誤貼回去讓 AI 修，不進下一個
```

**關鍵紀律**：
- 一次只做一個工單。貪多 AI 就會失憶、寫出接不起來的東西。
- 每個工單結束都要能「獨立測試通過」，不是「寫一半」。
- 不碰 LLM 的工單（1-5）先做完，把資料與狀態的地基打穩，再接 LLM。
- AI 若想改 CONTRACTS 的接口，要它先停下來問你。

## 給 AI 的對話開頭範本

做某個工單時，可以這樣開頭（範例為工單 13 story agent）：

```
我在實作一個 LLM 驅動的恐怖文字冒險遊戲。現在做「工單 13：story agent + 串流」。

【不可變接口，不准改名或改簽名】
（貼 CONTRACTS.md 的 DecisionPoint/BeatMeta/Option 資料類 + StreamParser 介面 + DELIM 常數）

【這個工單要做什麼】
（貼 BUILD-PLAN.md 工單 13 全文）

【story agent 的 prompt 規格】
（貼 skills/story/SKILL.md）

【實作時必須遵守的檢查項】
（貼 CHECKLIST.md 的 C2/C3/C4/C9 + E2 防暴雷）

請先寫測試（tests/test_story.py），再寫實作（core/agents/story.py）。
測試要包含「防暴雷斷言」：story 輸出不得含未 revealed 的 fragment content。
若你覺得接口有問題，先停下來問我，不要自己改。
```

這個範本的精神：**接口 + 工單 + prompt + 檢查項，四樣一起給**，AI 就有足夠上下文寫對，又不會因為看到整個專案而迷失。

## 進度追蹤建議

開一個 `PROGRESS.md`（你自己維護），記每個工單的狀態：

```
- [x] 工單 1 資料類 ✓ 2024-XX-XX
- [x] 工單 2 常數+SignalBus ✓
- [ ] 工單 3 Blackboard+patch ← 進行中
- [ ] 工單 4 ...
```

這樣你隨時知道做到哪，AI 也能從這份快速進入狀況。

---

## 最後提醒

MVP-A 是 19 個工單。看起來多，但每個都小、可測、有順序。**不要跳著做**——依賴關係是設計過的（資料類→狀態→LLM→agent→迴圈→前端）。照順序走，你會一直有「能跑的東西」，而不是寫到一半全部接不起來。

跑穩 MVP-A（30 beat 不崩、防暴雷、JSON 穩、存讀檔對），你的專題就成立了。MVP-B 和後續再依樣拆工單。
