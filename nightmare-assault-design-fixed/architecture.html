Nightmare Assault
多代理架構導覽

一款由 多個 LLM agent 協同驅動的無限恐怖文字冒險。玩家給一個主題，系統生出有「真相」的世界，再以分鏡（beat）為單位把故事無限說下去——同時擁有 玩家自由 與 劇情連貫。

8 個 agent 後端 Python · 前端 Web · pywebview OpenRouter 多模型 + Progress Kernel · Config Center

本簡報含兩個關鍵補丁：Narrative Progress Kernel（世界狀態權威）與 Config Center（可配置 prompt）。

核心命題

世界真相，不是劇情骨架

不可能三角：高自由度、劇情連貫、預定義骨架，三者最多同時要兩個。

本作放棄「預定義骨架」，改用世界真相——定義「發生過什麼、威脅是什麼、什麼致命、有哪些秘密可被發現」，但不定義事件順序。玩家怎麼跑都自由，而他發現的每個碎片自動互相吻合，因為它們同源。

▍ 類型定性 逃生即推理：威脅給壓力、資訊給出路。結局條件 = 資訊拼圖的完成。

agent 編制

8 個 agent · 各司其職

🧬 setup Heavy

開新局跑一次。生雙層 bible + NPC + 場景 + 開場。唯一不可降級。

🗝 orchestrator Light

每 beat。揭露閘門：把碎片 real→revealed。多數條件程式碼判。

✒ story Medium

每 beat、唯一串流。寫到決策點停筆。只讀 revealed_bible → 結構上不可能暴雷。

⚖ warden Light

每 beat、僅玩家。致命規則 + 結局 gate + 技能封頂。本地硬規則優先。

💬 npc-chat Light

聊天室開啟時。多輪對話、NPC 間互動。串流。

🌙 dreaming Light

非同步、每 K beat。在場 NPC 反思，更新演化層（無 warden 閘門）。

🚪 offstage-fate Light

非同步。離場 NPC 命運（機遇/失蹤/屍體/敵對），程式碼擲骰 + LLM 寫血肉。

🗜 compactor Medium

非同步、使用率門檻。滾動摘要 + ledger + 聊天封存濃縮。

每 beat 都跑且串流：只有 story 回合外非同步：dreaming · compactor · offstage-fate（玩家讀字時跑，不卡互動）
結構性防暴雷

雙層 Bible + 權限模型

story 暴雷的根因是「它看得到全部真相，只能靠自律不講」。雙層 bible 把它變成結構保證。

🔒 real_bible 鎖

完整真相：what_really_happened、deadly_rule、revelation_pool（碎片池）、結局條件。只有 orchestrator + warden + setup 能讀。

↓ 揭露閘門：達標才搬碎片 ↓

👁 revealed_bible 已揭露

已揭露子集。story 只能讀這個 → 沒揭露的它根本看不到，沒東西可暴雷。

補丁一 · 穩定化

Narrative Progress Kernel — 世界狀態權威

問題：讓 LLM 自由決定「世界狀態」，會出現開門後又問要不要開門、NPC 不進場、線索/背包不持久。

解法：把「推進判定」從 LLM 手上拿走，交給程式碼層的 kernel。story 退化成純 realizer——只把已決定的結果寫成敘事。

工程基礎

模型分層 · Fallback · 並行控制

三層模型 + fallback 鏈

層用途fallback
Heavysetup（真相密度高）sonnet → 3.5 → flash
Mediumstory · compactorflash → haiku → mini

▍ warden 的致命規則可在純 Python 保底——LLM 全失效也不誤殺、不全崩（graceful degradation）。

Blackboard 並行控制

版本化容器：每次 merge → version +1。

非同步只產 patch：dreaming/compactor 不直接寫，主迴圈在安全點驗 base_version 後才 merge；過期則拒/rebase。

同步路徑讀穩定快照：玩家搶快也不會讓 story 讀到半套狀態。

快照存「當時」摘要：回 beat 10 不會帶 beat 30 的摘要。

MVP-B · 差異化機制

讓世界活起來、讓一場遊戲有終點

🧠 UB4 輕量 dreaming

在場 NPC 每 5 beat 演化（情緒/意圖/關係）。只寫 evolving（碰不到 secret_core）；self_aware=false 不編謊；沒戲份凍結；非同步只產 patch。

🏁 UB2 一種結局序列

純敘述收尾 + 完整真相揭露（結局是唯一可露 real_bible 的時機）+ 復盤。提早觸發 → 沒找到的碎片一次攤開。

⚖ UB1 技能封頂強化

破格宣稱接受但加侷限（侷限變謎題/線索）。本地偵測，LLM 掛也封頂；(技能,侷限) 寫 ledger。

🎒 UB5 道具庫完整

held_by 綁 NPC（命運跟著走）、增刪查轉移、隨快照保存；is_key_item 永不外露。

🛡 UB3 防暴雷驗證報告

對抗性 caller（逐字 echo context）+ injection probes → 證明 story context 結構上無 real_bible。產驗證報告。

✅ 47 工單全 done

MVP-A + Progress Kernel + Config Center + MVP-B。555 測試綠。真實 LLM 實測連續生成、0 fallback。

▍ MVP-B 在穩定引擎上補差異化——NPC 會反思演化、一場遊戲能收尾並揭曉真相、破格宣稱被合理封頂。
整體圖景

多代理如何一起運作 — 一句話

setup 一次建好有真相的世界 → 每個 beat 由 warden 守門、kernel 決定世界怎麼推進、story 把結果串流成恐怖敘事（只讀已揭露、結構上不暴雷）、orchestrator 適時揭露新碎片 → validator 確保每步有後果 → compactor / dreaming 在回合外悄悄維護記憶與活的 NPC。所有人透過 Blackboard 溝通，沒人直接呼叫彼此。
