視覺化架構與表格參考

把 00–11 設計文件的核心結構濃縮成可視化圖表：6 張架構圖 + 5 張資料表。涵蓋 8 個 LLM agent 如何透過 Blackboard 協同、每個 beat 的資料流，以及兩個補丁（Progress Kernel、Config Center）。

State 層不依賴 LLM，可獨立測試。Agent 層透過 Blackboard 讀寫，不直接互相呼叫；錨點不可變性靠程式碼權限強制，而非 prompt 自律。兩個補丁是旁路增量層，預設可關、失敗降級。

圖 2

8-Agent 拓撲 — 一切透過 Blackboard

Heavy（setup） Medium（story · compactor） Light（其餘） 非同步（回合外，只產 patch） ★ 每 beat 都跑且串流

所有 agent 只跟中央 Blackboard 讀寫，彼此從不直接呼叫——這是並行安全與解耦的基礎。非同步 agent（dreaming/compactor/offstage-fate）在玩家讀字時跑、只產 patch，主迴圈在安全點驗版本後才 merge。

圖 3

Beat 迴圈時序 — 一個分鏡的資料流

同步路徑（紅）：玩家 → warden → kernel 決定推進 → story 只 realize（讀穩定快照、無 real_bible）→ validator 強制每步有後果 → 落 SQLite。非同步路徑（橘虛線）：壓縮與 NPC 演化在回合外只產 patch，安全點才 merge，不污染 story 讀取。

圖 4

雙層 Bible 與揭露閘門 — 結構性防暴雷

story 暴雷的根因是「看得到全部真相、只能靠自律不講」。雙層 bible 把它變成結構保證：完整真相鎖在 real_bible（只有 orchestrator/warden/setup 能讀），story 只讀 orchestrator 搬過來的 revealed_bible 子集——沒揭露的它根本看不到。

Agent 編制

Agent模型層頻率串流讀寫職責
setupHeavy開局一次否主題全部（初始化）生雙層 bible + NPC + 場景 + 開場。不可降級
orchestratorLight每 beat否real_biblerevealed_bible揭露閘門：碎片 real→revealed（多數程式碼判）
storyMedium每 beat是revealed_bibleBeatHistory · narrative寫 beat、停決策點。只讀 revealed → 不可能暴雷
wardenLight每 beat（玩家）否real_bibleverdict · ledger致命規則 + 結局 gate + 技能封頂（僅玩家）

dreamingLight非同步/每 K否ledger · evolvingnpc_evolving在場 NPC 反思演化（無 warden 閘門）

唯一每 beat 都跑且串流的生成 agent 是 story。dreaming/compactor/offstage-fate 為回合外非同步（玩家讀字時跑，不卡互動）。

錨點不可變性靠程式碼權限強制，而非 warden。玩家 → warden；NPC → 權限邊界，兩道機制各管不同對象、不交叉。違規寫 anchor 直接拋 PermissionError。
